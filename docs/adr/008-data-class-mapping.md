# ADR-008: Data class mapping — regular class vs C# record

## Status

Accepted

## Context

Kotlin `data class` generates: `equals()`, `hashCode()`, `toString()`, `copy()`, and `componentN()`. We need to decide how to represent this in C#.

C# has `record class` which gives similar features natively. But our bridge uses opaque pointers (`IntPtr _handle`) with explicit lifetime management (`IDisposable`).

## Alternatives Considered

### 1. C# `record class` with IDisposable

```csharp
public record Toy(string Name, string Color) : IDisposable
{
    internal IntPtr _handle;
    public void Dispose() { ... }
}
```

**Pros:**
- `Equals`/`GetHashCode`/`ToString` generated by C# compiler — no bridge calls needed
- `with` expression gives copy-with-modification syntax: `toy with { Name = "Ball" }`

**Cons — critical:**
- `with` creates a **shallow memberwise copy** on the C# side. This copies `_handle` without creating a new `StableRef`. Two C# objects now share the same handle. Disposing one invalidates the other — **use-after-free bug**.
- Record equality compares all fields including `_handle`. Two different Kotlin objects with equal values would have different handles → equality breaks.
- To fix: override `Equals`, `GetHashCode`, `ToString`, and `Clone()` — at which point we've overridden everything records give us for free.

### 2. Regular class + delegate to Kotlin (current implementation)

```csharp
public class Toy : IDisposable
{
    private IntPtr _handle;
    public override bool Equals(object? obj) => Native_Equals(_handle, ((Toy)obj)._handle);
    public override int GetHashCode() => Native_HashCode(_handle);
    public override string ToString() => Marshal.PtrToStringUTF8(Native_ToString(_handle))!;
    public Toy Copy(string name, string color) => new Toy(Native_Copy(_handle, name, color));
}
```

**Pros:**
- Safe — no accidental shallow copies of handles
- Equality semantics match Kotlin exactly (delegates across bridge)
- `Copy()` creates a proper new `StableRef` on the Kotlin side
- Consistent with how we handle regular classes

**Cons:**
- No `with` expression — must use explicit `Copy()` with all params
- More verbose than records
- `Copy()` requires all parameters (C# can't replicate Kotlin's default-to-current-value pattern)

### 3. Regular class + named optional params on Copy

```csharp
public Toy Copy(string? name = null, string? color = null)
{
    return new Toy(Native_Copy(_handle, name ?? this.Name, color ?? this.Color));
}
```

**Pros:** Approximates Kotlin's `copy(name = "new")` with `toy.Copy(name: "new")`
**Cons:** Uses `null` as sentinel for "keep current" which breaks if `null` is a valid value. Only works for nullable params.

## Decision

Use **option 2: regular class + delegate to Kotlin** with explicit `Copy()`.

- `Equals`/`GetHashCode`/`ToString` delegate across the bridge to Kotlin's generated implementations
- `Copy()` takes all constructor parameters, calls Kotlin's `copy()`, returns a new handle
- No `with` expression — the safety risk of shallow handle copies outweighs the ergonomic benefit

### Why not records

The `with` expression is fundamentally incompatible with opaque pointer bridges. It performs memberwise clone at the C# level without any hook to create a new `StableRef`. This leads to double-dispose and use-after-free. Overriding `<Clone>$()` is possible but undocumented/unsupported.

## Consequences

- Data classes use the same C# `class` + `IDisposable` pattern as regular classes
- Additional bridge methods: `equals`, `hashcode`, `tostring`, `copy`
- Consumer uses `toy.Copy("Ball", "Red")` instead of `toy with { Name = "Ball" }`
- Value equality works correctly across the bridge (not identity-based)
