- [053](053-nullable-reference-types-in-kotlin.md) – Phase 9 C# nullable reference types in Kotlin (Accepted): `NullableAttribute` (0=oblivious/1=non-null/2=nullable) + `NullableContextAttribute` decoded from ECMA-335 via `MetadataReader`, fallback chain member → method → type → oblivious; **oblivious binds as non-null `T`** (platform types `T!` are unspellable in generated Kotlin source, so the permissive middle every other interop layer uses is unavailable; non-null + a fail-fast bridge guard is its closest expressible approximation, matches C#'s own no-warning-on-oblivious behaviour, and preserves today's non-null string bindings), with an `info_oblivious_nullability` diagnostic (once per legacy assembly, per member for a `#nullable disable` island); nullability encoded as a `nullable` flag on the reference-typed `RirTypeRef` variants (not on the parameter/property carrier, which cannot express `List<string?>` when generics land); corrects ADR-051 (object returns are `Foo?` only when annotated) and unblocks the settable handle property (`var bar: Foo`), since a C# property carries one annotation shared by getter and setter; prerequisite fix: parameter names must key on `Parameter.SequenceNumber` (a nullable return adds a `Sequence == 0` `Param` row and shifts every positional lookup); `Nullable<T>` value types deferred (a closed generic struct, needs its own wire format)
- [054](054-reverse-bridge-registration-observability.md) – Tooling: reverse-bridge registration observability (Proposed): the ADR-041 function-pointer table becomes observable via three parts, deliberately not one. (1) An always-on generated `NugetRegistry` bakes the expected registration-export set in at generation time and records each one as it lands, so the `requireNotNull` guard's message becomes *computed*: "0 of 7 registrations have fired, nothing registered at all, the shims are not compiled into the host assembly (usually a stale `obj/project.assets.json`)" versus "6 of 7 fired, only `Sample.Text.Template` is missing". Those are the two sentences nobody could get an answer to during ADR-053. (2) An always-on registration-time contract self-check: every register export gains two leading scalars (`slotCount: Int`, `contractHash: Long`, both from one shared `RirBridging.kt` function) and its pointer params become nullable; Kotlin refuses to store *any* pointer on mismatch, turning stale-shim/fresh-dylib arity or type drift from silent memory corruption into a named "one of these is stale" error (an amendment to ADR-048's registration contract). (3) An opt-in `NUGET_INTEROP_TRACE=1` registration trace, stderr by default with a `NUGET_INTEROP_TRACEFILE` redirect (the `LD_DEBUG`/`LD_DEBUG_OUTPUT` and `DOTNET_HOST_TRACE`/`DOTNET_HOST_TRACEFILE` shape; xunit 2.9.3 does not capture Console, and a crashed host loses it). Registration granularity only, matching JNI's `-verbose:jni`; per-call tracing, verbosity levels, subsystem categories and an `EventSource` provider all rejected for v1 (EventSource is C#-only, so the Kotlin half of the trace could not participate). Zero cost on the bridge call path
