## Footguns

- `SampleApp.Tests.csproj` must set `<RuntimeIdentifier>` to `$(NETCoreSdkRuntimeIdentifier)`, never a hardcoded RID. CI runs on macos-latest (osx-arm64); a hardcoded `win-x64` builds fine but aborts `dotnet test` with "Could not find 'dotnet' host for the 'X64' architecture".
- Fixture packages are no longer pinned at `1.0.0`. Each pack mints a fresh `1.0.0-fixture.<epoch-ms>` and writes `sample-app/build/FixtureVersions.props`, which every fixture consumer imports. That is what keeps a re-pack from silently resolving the previous package's contents, so you do **not** need to purge `~/.nuget/packages/` or clear consumer `obj`/`bin` by hand. `scripts/verify-fixture-package-versioning.sh` guards this.
- That protection covers the repository fixtures only. Any package whose contents change **without** a version change can still be served from NuGet's cache. If you pin a version by hand in a manual test, you own the staleness.

## Debugging a failing reverse round trip

Get evidence before forming a theory. Two of the four "bugs" in the ADR-053 hunt were phantoms of stale build state, and a plausible story about freshly-changed code cost a 45-minute bisect that changed nothing.

Ask the cheap decisive question first, in this order:

1. **Read the failure message.** The "bindings are not registered" error is computed, not constant (ADR-054). It says *N of M registrations fired* and names the missing types, then ranks the likely causes. "0 of M" (nothing registered, the shim source probably never compiled in) is a completely different bug from "M-1 of M" (one type's shim or its `[ModuleInitializer]` failed). The message tells you which one you have.
2. **Run with `NUGET_INTEROP_TRACE=1`** (`NUGET_INTEROP_TRACEFILE=<path>` to redirect). It writes to **stderr** by default, because xunit v2 does not capture `Console.Out`. Both sides interleave, so "the C# shim never ran" and "the shim ran and the native side rejected it" look visibly different.
3. A **registration contract mismatch** (`slotCount` / `contractHash`) at startup is not a bug to debug. It means the C# shim and the native library came from different builds. Purge and rebuild; that check exists to tell you exactly that.

Only once those say nothing useful should you reach for a bisect.
