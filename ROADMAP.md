## Phase 2: KSP-driven generation
- [x] KSP processor that discovers all public declarations with full type info
- [x] **There is no export set: the processor bridges every public declaration it can see.** `NugetProcessor.process()` takes `resolver.getAllFiles()` with no filter, which is the line above meant literally — *all* public declarations. The `rootPackage` in `nuget { publish { } }` looks like the scoping knob and is not one: it only ever feeds `mapPackageToNamespace` (Phase 3), so it renames the C# namespace while the declaration set stays global, and that function even has an explicit branch that maps a package *outside* `rootPackage` anyway. Applying the plugin to a module that also holds unrelated public API therefore tries to bridge that API too, and any construct the forward direction cannot express there becomes a build break in generated code the user never wrote. This is MF-001 (declaration-level export scoping): an include/exclude mechanism, an annotation, or an effective package filter. First consumer to hit it was NYTimes-KMP (its BUG-003), which pays for it with a dedicated `:app:windows` export module whose `build.gradle.kts` carries a `TODO` explaining why the module exists at all. Verified directly: `getAllFiles()` and every `rootPackage` use site were read in source. Prior art for the decision is Swift Export and ObjC Export, which both had to answer "which declarations leave the module". **Done:** [ADR-063](docs/adr/063-forward-declaration-level-export-scoping.md) adds `publish { include(...); exclude(...) }`, mirroring `bind { include/exclude }` verb-for-verb (exclude wins, prefix match); an empty `include` defaults to `rootPackage` when set, so `rootPackage` now scopes as well as renames (a deliberate, documented behaviour change). A reverse-bound package (from `dependencies { dependency(...) { bind { } } }`) is always exported regardless of `include`/`exclude`, since `test-library`'s own forward/reverse round trip broke on the first cut of this feature when a bound package was dropped from scope.
- [x] **A type declared in a dependency module is never exported, so a method or `Flow<T>` returning one has no usable C# model.** The same missing idea as the item above, seen from the other side: `getAllFiles()` returns the *module's own* sources, so a referenced type one module away is never translated, while an unrelated type in the same module is. Consumers must hand-write a module-local projection DTO plus mapper functions for every shared model they want to surface, which is what NYTimes-KMP does today (its BUG-004, MF-003) for `Flow<TopStoriesState>` where `TopStoriesState` lives in `:app:presentation`. Both items resolve to defining the export set as a reachability closure from explicit roots rather than "the files in this module", so whichever lands first should choose that framing rather than patch its own half. **Sequencing:** this must not land before the forward diagnostics item (Tooling & Test Integrity) and the value-class-parameter item (Phase 4), or reachability will drag the exact constructs module isolation currently hides — a `CharSequence`-delegating value class, a generic suspend extension — back into the export set and re-arm them. **Both prerequisites are now satisfied**: [ADR-064](docs/adr/064-forward-unsupported-declaration-diagnostics.md) names both named constructs (`SKIPPED_INHERITED_MEMBER`, `SKIPPED_UNSUPPORTED_COMBINATION`) rather than letting them re-arm silently, and the value-class-parameter item closed under [ADR-062](docs/adr/062-forward-callable-plan.md) Phase 9. Verified directly (`getAllFiles()` read in source); cut from [MVP.md](MVP.md). **Done:** [ADR-066](docs/adr/066-forward-export-reachability-closure.md) defines the export set as a reachability closure from module roots (returns, parameters, property types, type arguments of admitted carriers, sealed subclasses, primary-ctor params), admitted by ADR-063's existing package predicate, no new DSL verb. Ships with two forced corrections to ADR-064, landed in the same feature because reachability is what makes both re-arm: the `SKIPPED_INHERITED_MEMBER` filter's `origin != Origin.KOTLIN` test is replaced by an origin-independent supertype simple-name match (every member of a cross-module declaration reports `origin == KOTLIN_LIB`, which the old rule would have read as "everything inherited"), and every `Modifier.VALUE`-only classification site becomes `VALUE || INLINE` (a cross-module `value class` reports `INLINE`, never `VALUE`). Two new diagnostics: `SKIPPED_UNEXPORTED_DEPENDENCY_TYPE` (reachable but out-of-scope, names the `include(...)` fix) and `INFO_EXPORTED_FROM_DEPENDENCY` (one aggregate manifest line per KSP run). Fixture: a real second Gradle module, `:test-models`, consumed by `:test-library` via `implementation(project(":test-models"))`, crossing every mechanism (`Newsroom.kt` + `IntegrationTests/NewsroomReachabilityTests.cs`). Corrected in the ADR itself while landing this: the ADR's original claim of five `Flow<T>` element-type-by-simple-name call sites was wrong; only two are the `Flow` bug (`CirClassTranslator.kt:149`/`:458`, now qualified `global::{namespace}.{Type}`), the other three share the same three-line shape for an unrelated, still-open `List`/`Set` element-type gap.
  - [ ] **Duplicate-type hazard across two published packages** (accepted limitation, decided with the human, not a defect to fix here). If two Gradle modules each publish forward and each independently admit the same dependency-module type through their own closure, the consumer gets two unrelated C# types for one Kotlin type (`PackageA.Models.Foo` and `PackageB.Models.Foo`), with no conversion and no diagnostic, because neither KSP run can see the other package's export set. Same hazard Kotlin/Native framework export already has when two frameworks each `export()` a shared dependency. [ADR-066](docs/adr/066-forward-export-reachability-closure.md)'s Consequences section sketches three shapes for a fix (a shared "models" package both publishers depend on instead of inlining; a diagnostic when an admitted type's origin module is itself published; an opt-out forcing such a type back to `SKIPPED_UNEXPORTED_DEPENDENCY_TYPE`), none decided.
  - [x] **Collection-element-unsupported crash, fixed alongside this feature**: `ForwardCallablePlanValidator.validateType` (`ForwardMarshallingModel.kt:353`) used to hard-error with an `IllegalStateException` on an unsupported collection element instead of skipping the one callable, so a single unsupported `List`/`Map`/`Set` element type killed the entire `packNuget` run. Predated this feature; the `:test-models` cross-module fixture was the first thing to exercise the path (an admitted dependency type nested inside a collection). Fixed via a new `isBridgeableComponent()` gate on `BridgeType`, so an unsupported element now routes the whole callable through the normal named-skip diagnostic path instead of crashing generation; regression coverage in `Tier1CollectionElementSkipTest.kt`. Discovered alongside [ADR-066](docs/adr/066-forward-export-reachability-closure.md).

## Phase 8: Ecosystem — consuming NuGet packages from Kotlin

The inverse of everything above, modeled on the Kotlin CocoaPods plugin (`pod("...")` → cinterop → Kotlin externals): the Gradle plugin resolves a C# NuGet dependency, extracts its public API from the .NET assembly metadata, and generates Kotlin-idiomatic bindings so Kotlin code can call the C# library — and implement interfaces it defines. Because the Kotlin/Native library always runs inside a .NET host process, Kotlin → C# calls need no CLR hosting: the generated C# side registers function-pointer thunks with Kotlin at startup.   - [x] Lifetime cleanup: `Cleaner`-primary + explicit idempotent `close()` (`kotlin.AutoCloseable`) → shared `nuget_runtime_register` free export — deliberate inversion of the forward `IDisposable`-first choice (see [ADR-051](docs/adr/051-csharp-objects-as-opaque-handles.md))
- [x] Map instance constructors (`new Foo(...)` → Kotlin secondary `constructor` delegating through a bridge helper; [ADR-057](docs/adr/057-csharp-overload-sets-in-kotlin.md) extends this to every bridgeable public constructor) (see [ADR-052](docs/adr/052-csharp-instance-constructors-in-kotlin.md))
- [x] Map instance methods and instance properties (v1-mappable parameter/return types first). No new ADR; a confirmed mirror of [ADR-051](docs/adr/051-csharp-objects-as-opaque-handles.md)'s stated principle that an instance thunk is a static thunk whose first parameter is the receiver handle. Read-only property → `val`, settable → `var` with bridge-backed `get()`/`set()`. Registration categories are constructors → static methods → instance methods → per-property getter then setter, canonically sorted as of [ADR-057](docs/adr/057-csharp-overload-sets-in-kotlin.md)
  - [x] Instance members whose Kotlin name would collide with the [ADR-051](docs/adr/051-csharp-objects-as-opaque-handles.md) wrapper's own `handle` / `close` / `cleaner` are skipped with a warning; statics are unaffected (they live in the `companion object`)
  - [ ] **A bound C# class with an `init`-only property generates a Kotlin setter whose C# thunk does not compile.** The metadata reader reports an `init` accessor as an ordinary settable property (`RirProperty.isReadOnly = false`), so the generators emit a Kotlin `var` and a C# setter thunk body `target.Prop = value;`, which is `CS8852` (an `init`-only property can only be assigned in an object initializer). No shipped fixture has an `init`-only property on a bound *class* (only on a Shape B *struct*, where reconstruction already uses an object initializer and the bug doesn't apply), so this is latent, not currently breaking any build; unverified against a repo fixture, verified only via a scratch cross-assembly compile. Root cause and fix are known: detect the `IsExternalInit` required modifier on the setter's signature (its metadata shape is verified to resolve, see [ADR-058](docs/adr/058-csharp-shape-b-structs-in-kotlin.md) Constraint 1.5) and report such a property as read-only (or skip its setter) instead of settable. Discovered alongside [ADR-058](docs/adr/058-csharp-shape-b-structs-in-kotlin.md); its Shape B reconstruction needs no such detection, so this ADR does not fix it
- [x] Map static C# properties (getter/setter, v1-mappable types) — straightforward extension of the same function-pointer pattern (see [ADR-048](docs/adr/048-kotlin-stub-generation-from-reverse-ir.md)); static classes expose them on a Kotlin `object`, non-static classes expose them on the `companion object`
- [x] Map C# enums → Kotlin `enum class` (v1: public top-level, default-`int`, non-`[Flags]`, unique contiguous `0..N-1` values; mirror of [ADR-006](docs/adr/006-enum-mapping.md))
- [x] Map nullable reference type annotations → Kotlin `T?` (`NullableAttribute`/`NullableContextAttribute` in metadata; oblivious/un-annotated legacy assemblies bind non-null with a fail-fast bridge guard, plus an `info_oblivious_nullability` diagnostic) (see [ADR-053](docs/adr/053-nullable-reference-types-in-kotlin.md))
  - [x] Setter for handle-typed properties: a C# `Foo Bar { get; set; }` now renders as a Kotlin `var`, because a property carries exactly one `NullableAttribute` so getter and setter agree on one type. Unblocked by ADR-053
  - [ ] `Nullable<T>` value types (`int?`) are a different, deferred feature (ADR-053 Decision 3): no `NullableAttribute` machinery applies. **Unblocked** by [ADR-056](docs/adr/056-csharp-structs-in-kotlin.md): its struct out-pointer convention (`void` + one out-pointer per component, one crossing, one evaluation) is exactly the wire format this item was waiting on, so it needs no further ADR, only the reader work to stop `GetGenericInstantiation` dropping `System.Nullable<T>`, plus the generator work to emit the `byte hasValue` + out-pointer shape
- [x] Map C# structs (value types): **not** blittable pass-by-value vs boxed handle, the framing this line originally used. Both were dead ends: pass-by-value is not expressible in this project's architecture (a hand-written `CStructVar` cannot be used in a Kotlin/Native `CFunction`, only a cinterop-generated one, which would need a per-target build stage this project deliberately doesn't have), and a boxed `GCHandle` destroys value semantics (`close()`, `Cleaner`, reference equality on what should be a value type). The actual answer is a third option neither name named: **component decomposition**, no struct ever crosses the wire. A struct parameter expands to one ABI argument per component; a struct return expands to `void` + one out-pointer per component. The Kotlin surface is an immutable `data class`: value equality, `copy()`, destructuring, no handle, and no `close()`. A Shape A struct's unique state-covering constructor costs zero registration slots; [ADR-057](docs/adr/057-csharp-overload-sets-in-kotlin.md) gives each alternate public constructor one bridge-backed slot. Supported components are primitives, `string`, and bound enums on static and instance members, including struct-typed properties (see [ADR-056](docs/adr/056-csharp-structs-in-kotlin.md))
  - [x] Struct methods and computed properties: the direct mirror of ADR-014's "reconstruct the value class on each invocation"; the struct gets its own registration export when it has bridgeable members (or alternate constructors). Reader enumerates public non-void instance methods, get-only non-component computed properties, and static methods; skips Equals/GetHashCode/ToString/Deconstruct, operators, setters, void instance methods, and component auto-properties. Wire: N component args first (reconstruct on call), ordinary args after; struct returns via existing out-pointers. Slot order: alternate ctors → static methods → instance methods → computed getters (see [ADR-056](docs/adr/056-csharp-structs-in-kotlin.md) Scope follow-up, [ADR-014](docs/adr/014-value-class-mapping.md))
  - [x] Shape B structs: no public constructor; public settable fields and/or settable auto-properties (`set` or `init`) instead. The metadata reader now extracts fields (`TypeDefinition.GetFields()` + backing-field attribution), recovering components and their order from FieldDef declaration order, unified across fields and auto-properties. C# reconstructs with an object initializer (`new T { A = a, B = b }`) instead of a constructor call; the wire format, the immutable `data class` surface, and the struct-members machinery (methods, computed properties, statics) are unchanged from Shape A. A Shape B struct binds no constructors at all, alternate or otherwise (diagnosed with `skipped_unsupported_struct`, not silently dropped). Also fixed as part of this: the ADR-054 contract hash for a struct reference now hashes component **names** as well as types, closing a latent hole where reordering two same-typed components was invisible to the stale-shim check (see [ADR-058](docs/adr/058-csharp-shape-b-structs-in-kotlin.md))
    - [ ] Alternate constructors on a Shape B struct: diagnosed, not bound, since the struct's object-initializer primary already reaches every component and an alternate would collide with it. Deferred by [ADR-058](docs/adr/058-csharp-shape-b-structs-in-kotlin.md) Decision 4a; needs a rule for the primary-constructor collision before it can bind
    - [ ] Manual (non-auto) settable properties as Shape B components (a hand-written private field behind a settable property, e.g. `struct Manual { private int _a; public int A { get => _a; set => _a = value; } }`): metadata alone cannot prove the setter writes the field, so such a struct is skipped rather than bound. Deferred by [ADR-058](docs/adr/058-csharp-shape-b-structs-in-kotlin.md) Decision 2a; would need IL analysis or a heuristic
    - [ ] Shape A constructor-parameter nullability decoding: a pre-existing ADR-056 gap (a struct's state-constructor parameters are not nullability-decoded, unlike its properties). Surfaced but not closed by [ADR-058](docs/adr/058-csharp-shape-b-structs-in-kotlin.md), whose own Shape B components *are* nullability-decoded. Widened, not closed, by [ADR-059](docs/adr/059-nested-struct-components-in-kotlin.md): nesting a Shape B inner (nullability-decoded) inside a Shape A outer (not) mixes decoded and undecoded components in one flat argument list: not incorrect (a `string?` component still binds non-null and fails fast rather than corrupting), just confusing to reason about. Deliberately kept out of ADR-059's scope, since it's orthogonal to nesting
  - [x] Nested struct components: a struct-typed component may itself be a struct, at any depth. The wire flattens recursively, depth-first pre-order: the flat ABI arg list is the leaves of the component tree, and the ABI name of each leaf is its `_`-joined path. The Kotlin surface stays nested (`data class` inside `data class`), so deep structural equality and a composed `copy()` fall out for free. C# reconstruction is recursive: the outer struct's shape decides how a component is *placed*, the inner struct's shape decides how it is *built*, so all four Shape A/B nesting combinations (A-in-A, B-in-A, A-in-B, B-in-B) work with no rules of their own, at any depth and across a Kotlin package boundary. Depth-1 output is byte-for-byte unchanged. A member whose flattened ABI arity (receiver + parameters + out-pointers) exceeds Kotlin/Native's 22-argument `CFunction.invoke` ceiling is skipped, not generated, with a new `skipped_abi_arity_limit` diagnostic naming the member and the count; the rest of the type still binds. Fixed as part of this, each load-bearing (nothing bound without it): the metadata reader's structural-equality check gained the `RirStructType` case it was missing (every struct-typed constructor-parameter-vs-property match had failed closed since ADR-056, unreachable until a struct had a struct component); the Kotlin bindings generator now imports a nested component's type instead of emitting a bare unqualified name; the same generator now also collects STRUCTS reached through nesting for cross-package imports, not just enums; and the reader's Shape A/Shape B double-failure diagnostic no longer discards the Shape A reason, so a nested Shape A component that fails no longer gets misreported as a Shape B auto-property failure (see [ADR-059](docs/adr/059-nested-struct-components-in-kotlin.md))
    - [ ] Raising the 22-argument ceiling: pass a pointer to a packed scratch buffer once a member's flattened arity exceeds 22, instead of skipping the member. Sketched, not decided: needs its own call on buffer ownership, layout, and alignment (deferred by [ADR-059](docs/adr/059-nested-struct-components-in-kotlin.md) Decision 5d)
    - [ ] `Nullable<T>` components, including a nullable *nested* struct (e.g. `Profile?` inside `Litter`): the existing struct out-pointer + `byte hasValue` format (see the `Nullable<T>` value types item above) extends to it without a new wire format, but it stays its own ROADMAP slice (deferred by [ADR-059](docs/adr/059-nested-struct-components-in-kotlin.md) Scope)
  - [ ] Class-typed (handle) components inside a struct: deferred on semantics, not cost. A `GCHandle` inside a value type does not compose, since `copy()`ing the Kotlin `data class` would duplicate a `Cleaner`-managed handle. Needs its own decision (ADR-056 Scope). Confirmed unaffected by nesting: a handle component would be a **leaf** on the wire (an `IntPtr`), so the flattening recursion never descends through it either way (see [ADR-059](docs/adr/059-nested-struct-components-in-kotlin.md) Scope)
- [x] Overload sets: preserve ordinary same-name Kotlin overloads while deriving stable internal bridge identities from canonical managed signatures. Supported siblings bind independently, true mapped-Kotlin-signature collisions fail generation, and Shape A structs may expose bridge-backed alternate constructors while their unique state constructor remains slot-free (see [ADR-057](docs/adr/057-csharp-overload-sets-in-kotlin.md))

## Post-migration hardening

Follow-up work from the completed [forward marshalling centralization migration](MIGRATION.md). The clean full verifier passes, but review found four edge cases outside the current fixture coverage.

- [ ] **Surface ordinary planner skips as generation-time diagnostics.** `ForwardCallablePlanCatalog` retains `Skipped` entries, but emission consumes only successful plans, so unsupported ordinary public APIs can disappear silently. Report every non-specialized skip through KSP with the declaration, position/role, classified type, and reason; keep named specialized protocols explicitly exempt. Add a real KSP fixture proving an unsupported declaration fails loudly instead of disappearing.
- [ ] **Preserve getters for mutable collection properties.** `ForwardPropertyPlanner` currently rejects an entire property when a Kotlin `var` has a collection type, regressing the previously generated read-only C# getter. Plan getter and setter independently: retain the working getter, then either plan the setter when input conversion exists or emit a setter-specific diagnostic. Add regression coverage for the supported declaration positions.
- [ ] **Make list-input eligibility match element marshalling.** Every `List<T>`/`MutableList<T>` input, while `NugetMarshal.CreateList<T>` can box only a subset, so admitted element types such as `Byte`, `Short`, unsigned primitives, `Char`, enums, and nullable elements can fail at runtime. Compose element lowering from `BridgeType`, or reject unsupported element types during planning with a named diagnostic; add consumer coverage for every admitted element category.
- [ ] **Dispose temporary list handles on every exit path.** Generated C# checks and throws for a Kotlin error before disposing the temporary StableRef-backed input list, and `CreateList` can leak if enumeration or element conversion throws. Generate `try`/`finally` ownership scopes covering construction and invocation, and add exception-path tests that prove disposal.

## Tooling & Test Integrity

Fallout from [ADR-053](docs/adr/053-nullable-reference-types-in-kotlin.md) (reverse nullability), the first fixture to exercise the reverse bridge realistically. It flushed out four latent defects, **two of which were phantoms of stale build state**, and the debugging cost was dominated by having no way to observe the bridge and no way to trust the build. These items exist so the next feature does not pay that cost again.

**Rejected: a `scripts/verify.sh --fast` mode.** It was proposed on the theory that agents hand-patch generated output and the NuGet cache because the sanctioned path is too slow. Measurement kills that theory: verify is 38s clean-room after a source change, 18s warm. Nobody cheats to save 14 seconds. Agents did it because they were being sloppy, which is a discipline problem, and the `PreToolUse` guardrail now makes it impossible. Worse, the only real saving came from skipping `:test-library:clean`, which is precisely what produces a truncated `Interop.cs` (see the KSP bug below), so a fast mode would have handed agents a *sanctioned* way to manufacture the phantom bugs this whole section exists to prevent. Do not re-add it.

- [x] **Reverse-bridge registration observability: always-on self-check first, trace second.** The [ADR-041](docs/adr/041-kotlin-to-csharp-call-mechanism.md) function-pointer table was unobservable: nothing could report which `nuget_{ns}_{type}_register` exports actually fired, so "did registration work?" was only answerable by bisecting. The research found that an opt-in trace, the original framing of this item, is the weaker half of the fix: nobody enables a trace for a bug they don't already suspect, and in the ADR-053 hunt nobody suspected registration. What would actually have caught it is **always-on**, no env var required: every register export now carries a registration-time contract self-check (`slotCount: Int` and `contractHash: Long`, amending [ADR-048](docs/adr/048-kotlin-stub-generation-from-reverse-ir.md)'s and [ADR-049](docs/adr/049-csharp-registration-shim-generation.md)'s registration contract), which refuses to store any pointer and fails loud, naming both sides, the instant a stale C# shim and a freshly built native library disagree; and a generated `NugetRegistry` that turns the old constant "bindings are not registered" message into a computed "N of M registrations fired", naming exactly which types are missing so a whole-assembly problem (nothing fired) is never confused with a single-type problem (everything but one fired). The opt-in `NUGET_INTEROP_TRACE=1` / `NUGET_INTEROP_TRACEFILE=<path>` trace (stderr by default, JNI/`LD_DEBUG`/`DYLD_PRINT_BINDINGS` precedent) ships alongside as the cheap complement for the case a consumer already suspects registration and needs ordering detail the two always-on checks can't express (see [ADR-054](docs/adr/054-reverse-bridge-registration-observability.md))
- [x] **Put the ranked remediation guidance into `NugetRegistry.notRegistered()`.** [ADR-054](docs/adr/054-reverse-bridge-registration-observability.md) now makes a zero-registration failure rank the likely causes: stale consumer build state, a missing package reference, then an unloaded shim assembly. It names the stale `obj/project.assets.json` failure mode and cleanup, while a partial result instead limits diagnosis to the missing `{Type}Registration.cs` shim or its `[ModuleInitializer]`.
- [x] **Stop pinning fixture packages at `1.0.0`.** Each fixture build gives `TestDependency` and `TestLibrary` an immutable `1.0.0-fixture.<epoch-ms>` version, then writes `build/FixtureVersions.props` for every fixture consumer to import. `TestLibrary` declares the same exact version of `TestDependency`, so both NuGet identities are fresh together. `scripts/verify-fixture-package-versioning.sh` proves two consecutive packs restore and build all consumers without clearing NuGet caches or consumer `obj`/`bin`.
- [x] **Forward-direction ABI contract check.** The KSP processor normalizes generated C# `CirDllImport` declarations and Kotlin `@CName` exports, then fails generation when an export is missing, duplicated, or has a mismatched result or ordered parameter contract. This catches forward source drift before packaging, with no generated API or runtime ABI change (see [ADR-055](docs/adr/055-forward-abi-contract-check.md)).
- [ ] **Extend the forward ABI contract check to the specialized legacy protocols.** The ADR-055 check above covers only ordinary callables. `ForwardAbiContract.csharp(file)` (`ForwardAbiContract.kt:90`) builds the C# side for a class from `ordinaryNativeImports()` (`CirNativeImports.kt:3`), which filters out `isAsync`/`isFlow` methods (`:14`) and `usesLegacyNativeImport()` properties (`:10`), and `assertMatchesPlan` only checks planned callables ([ADR-062](docs/adr/062-forward-callable-plan.md)). So the suspend, `Flow`, lambda/stored-callback/interface-bridge, sealed-helper and generic routes (exactly those enumerated in `ForwardAbiLegacyRoutes.kt`) sit in **neither** guard: their Kotlin `@CName` half and their C# `DllImport` half are built by fully independent code paths with no generation-time equality check between them. A divergence of the ADR-053 SIGBUS class (a C# import missing the `out IntPtr error` slot the Kotlin export declares) would ship undetected for those routes, where ordinary callables have been guarded since ADR-055. Fix: emit `ForwardAbiSignature`s for the legacy helper imports so `assertMatches` compares both halves even while they stay on independent code paths; the helper import shapes not fitting the current `ForwardAbiSignature` normalizer is the actual work. This is the cheap, urgent half of closing the legacy-route gap, ahead of any plan migration. Verified directly: the filters and both checks read in source. Discovered alongside the [ADR-062](docs/adr/062-forward-callable-plan.md) migration review
- [ ] **Route the composed inner types of the legacy protocols through `BridgeType`.** [ADR-062](docs/adr/062-forward-callable-plan.md) centralized ordinary-callable marshalling into one `BridgeType` classification plus plan, retiring the ADR-060 position-not-type duplication for those paths, but the specialized protocols still on legacy routes (`exports/` + `cir/`) do not consume it for the types they wrap: a `Flow<T>`'s element, a suspend return, a callback parameter each re-derive their own marshalling. So adding a new composed type (a new collection kind, a new primitive width) still means threading it through every protocol separately, the residual of the duplication the migration removed everywhere else. The cost is bounded (each protocol is a bespoke one-off, not a rule repeated across positions), so this is lower priority than the contract-check item above. Scope: keep each protocol *shell* on its legacy route and route only its composed inner types through `ForwardBridgeTypeClassifier` / the plan's marshallers. Full plan adapters for the shells themselves are the separate per-protocol follow-up ADR-062 names in its Consequences, done opportunistically when a feature next touches each protocol rather than as a standalone migration. Discovered alongside the [ADR-062](docs/adr/062-forward-callable-plan.md) migration review
- [ ] **Compile the generated Kotlin in the plugin unit tests.** `NugetGenerateBindingsTaskTest` and `NugetGenerateShimsTaskTest` assert on generated text with substring matching, so they cannot see whether the text they approve is *valid source*. The ADR-054 walking skeleton emitted literal backslashes into `NugetRegistry.kt` (backslash-escaped quotes inside a Kotlin raw string) and every unit test passed; only actually compiling the output caught it. The forward direction already has `GeneratedBindingsCheck`, which compiles the generated C# with warnings as errors. The reverse direction has no equivalent for the generated Kotlin. Add one, so a generator that emits syntactically broken source fails in seconds rather than at the end of a full round trip. The *forward* half is now covered: [ADR-060](docs/adr/060-adversarial-forward-fixture.md)'s Tier 1 harness **compiles** the generated forward Kotlin (`CNameExports.kt`) in-process rather than substring-matching it, which is exactly the blindness this item was written against — the ADR-054 walking skeleton's literal backslashes passed every substring test. The *reverse* generated Kotlin still has no compile check, which is what keeps this box open
- [x] **An adversarial reverse fixture.** Every reverse fixture before `Test.Nullability` had exactly **one bound class per namespace**, and no exported nullable-returning function ever threw. That is why three latent bugs (the metadata reader's positional parameter lookup, the colliding top-level registration vars, the missing `errorOut` on the forward two-call P/Invoke) sat green in CI. `Test.Household` in `TestDependency/Household.cs` now combines the hard cases in one namespace: three bound classes (`Cat`, `Toy`, `Household`), a settable handle-typed property (`Cat.FavoriteToy`), two types sharing a method name (`Cat.Describe()` / `Toy.Describe()`), a four-parameter method with a nullable annotated return (`Household.FindToy`, the shape that exposed the positional-parameter shift), and an oblivious `#nullable disable` island (`StrayCat`) whose method returns a genuine C# `null` so the ADR-053 fail-fast guard is exercised rather than just its happy path. Asserted by `HouseholdRoundTripTests.cs` through the ADR-050 round trip. It found no new bug, which is the point: the previously latent defects stay fixed under a harsher shape than the one that first exposed them. Note the fixture does **not** throw from a reverse-bound C# method: per [ADR-049](docs/adr/049-csharp-registration-shim-generation.md) thunks never catch, so that would fast-fail the host process rather than fail an assertion (reverse exception propagation is Phase 11). The forward *exported* nullable-throw path, which is where the `errorOut` bug actually lived, is covered instead
- [x] **An adversarial forward fixture.** The exact mirror of `Test.Household` above, needed for the same reason and overdue for longer. `test-library` is the only forward fixture corpus and it is *ours*: shaped by us, exercising features we already knew we had built. [NYTimes-KMP](https://github.com/xxfast/NYTimes-KMP) is the first forward consumer this project has that it did not write, and it hit eight defects on constructs marked `[x]` — six confirmed in source, filed above under Phases 2–6. `GeneratedBindingsCheck` (ADR-055) is not broken: it cannot fail on a shape `test-library` does not contain, and the holes are stark once looked for. `CatRegistry` is the only Kotlin `object` and never returns a `String`; not one value-class fixture has a method that takes a parameter. - [x] **Forward unsupported-declaration diagnostics.** The mirror of [ADR-043](docs/adr/043-bridgeable-subset-boundary.md), which gave the *reverse* direction a defined bridgeable subset and a diagnostic naming each skipped member and why. - [ ] **The forward processor has almost no unit-test seam.** `nuget-processor/src/test` holds two files (`CirFunctionTranslatorNullableTest`, `ForwardAbiContractTest`) against nine for the reverse `nuget-plugin`. Not a bug, a cost, and worth weighing before the Phase 2–6 items above are worked, since they are four codegen fixes in a row through that loop. Plausibly also part of why they went unseen: the reverse generators got unit-tested into existence, the forward ones were only ever observed through a full round trip. Note the complement already tracked below — the reverse direction lacks a *compile* check on its generated Kotlin, while the forward direction gets one free (its `CNameExports.kt` is compiled by `:test-library:compileKotlinMingwX64`), which is why the Phase 4 value-class and collection items surface as build breaks rather than silent text. **Partly closed by [ADR-060](docs/adr/060-adversarial-forward-fixture.md)'s Tier 1 harness**, which is exactly this fast inner loop for the slice it needs — "Kotlin source in → real `NugetProcessor` → generated `CNameExports.kt` compiled in-process", red in ~sub-second per cell after warmup — and is the loop each of the four upcoming forward codegen fixes runs through. - [ ] **KSP incremental processing silently drops suspend functions from `Interop.cs`.** `NugetProcessor.kt` partitions functions into `suspendFunctions` / `regularFunctions`, but omits `suspendFunctions` from **both** `hasNothingToProcess` and the `sources` array feeding `Dependencies(aggregating = true, *sources)`. The KSFiles holding suspend functions are therefore never declared as dependencies of the aggregating output, so KSP drops them on an incremental run. Verified repro: touch any `test-library` source, run `:test-library:packNuget` **without** `clean`, and the generated `Interop.cs` comes back missing the `AsyncFunctions` and `AsyncExceptions` classes; consumers then fail with `CS0103`. `clean` + `packNuget` regenerates all 37 classes correctly. **This is consumer-facing, not a test artifact**: the normal dev loop (edit one file, re-pack, no `clean`) ships a package whose bindings are missing every suspend function. Suspected second manifestation: a library whose only exports are suspend functions may generate nothing at all
- [ ] **The reverse interop output directory does not clean orphaned files from a prior run** (`NugetGenerateBindingsTask`/`NugetGenerateShimsTask`, `build/nuget-interop/`). The tasks write the current run's generated files but are reported not to clear the directory first, so a `.kt`/`.cs` file generated for a type later renamed or removed from a bound namespace can persist alongside the fresh output. Reported by `csharp-dev` while building the [ADR-056](docs/adr/056-csharp-structs-in-kotlin.md) struct fixture; **not independently verified here**. Check whether the tasks declare their output directory the standard Gradle way, which gives stale-output cleanup for free, or write directly and bypass it, before deciding this needs code and not just a report
- [x] **Let the `research` agent verify what it asserts.** [ADR-053](docs/adr/053-nullable-reference-types-in-kotlin.md) stated that `NullableAttribute`'s constructor is always a `MethodDefinitionHandle`. On net8.0 it is a `MemberReferenceHandle`, so an implementing agent followed the ADR literally and silently decoded every annotation as oblivious. Both halves are now closed. The agent has `Bash` and a **verification spike** protocol: scratch directory only (`mktemp -d`), never the repo, used to falsify the one claim the design actually rests on ("if this is wrong, does the implementation silently produce wrong output?"), and a claim is promoted from inferred to verified only by output it actually saw. And the walking skeleton's second job is now to falsify every claim still labelled `inferred` when it arrives, on the rule that if a claim does not survive contact with the artefact, **the ADR is wrong** and gets fixed, rather than the code being bent to match the doc
- [ ] **One fatal forward diagnostic still lives outside `ForwardDiagnosticKind`.** `CirFunctionTranslator.kt:91-99`'s `enumParamsUnsupported()` (a top-level function with enum parameters and a return shape that can't cast the enum ordinal back down) is a pre-existing `logger.error` fail-fast that predates and was deliberately left outside the [ADR-064](docs/adr/064-forward-unsupported-declaration-diagnostics.md) model: it isn't a signature collision, and it didn't fit cleanly into any of the seven named kinds. Small residual, not urgent; a future tidy-up is to give it its own `ERROR_*` kind so every fatal forward diagnostic routes through the one sink

## Future Improvements

- Support flat/unnested sealed class hierarchies (subclasses as top-level in same namespace)
- KSP incremental processing if build times become a concern on large libraries
- Map data classes to C# `record class` if a safe `with`-expression pattern can be found (see [ADR-008](docs/adr/008-data-class-mapping.md))
- Verify Kotlin GC actually frees objects after all StableRefs are disposed (requires Kotlin-side weak references + GC trigger — not feasible in standard unit tests)
- Memory leak detection tooling for bridged objects in CI
- Object identity preservation (caching wrappers) if profiling shows allocation overhead is significant
- Custom type mappers for dependency types (e.g., `kotlinx.datetime.Instant` → `DateTimeOffset`)
- Pure-JVM ECMA-335 metadata reader + NuGet v3 client, dropping the .NET SDK prerequisite from the Kotlin-side build (synthesis D1) — no plugin in this space runs prerequisite-free; a genuine ergonomic edge
- Hand-written C# shim escape hatch for API members outside the auto-bridgeable subset (spm4Kmp's bridge-folder model, synthesis D2)
- Forward un-skip/remap escape hatch: a Xamarin-`Metadata.xml`-style (or `@HiddenFromObjC`-style opt-out) mechanism to force-bind or remap a construct [ADR-064](docs/adr/064-forward-unsupported-declaration-diagnostics.md) would otherwise skip. Deferred by ADR-064's Scope; today the diagnostic's hint text names the hand-written-adapter workaround instead, the same choice ADR-043 made for reverse
- Reverse-bridge integration tests against real published NuGet packages (not just the controlled `TestDependency` fixture) — chosen precisely because they *don't* fit the bridgeable subset cleanly, to prove the ADR-043 skip diagnostics and partial-binding behaviour on messy real-world surfaces (mixed bridgeable/unbridgeable members, legacy nullability, multi-TFM)
- Map KDoc annotations to C# XML docs for better IDE support
- Expose Kotlin `Job` as a mapped C# type so cancellation can be tied to the job directly (e.g., `job.Cancel()`) instead of requiring a pre-created `CancellationTokenSource`
- NativeAOT compatibility for the generated bindings (see [ADR-038](docs/adr/038-aot-compilation.md)) — blocked on a reflection-free generics dispatch path; `[DllImport]` → `[LibraryImport]`; AOT publish smoke test in CI
- Centralize version-sensitive C# tokens into a `CSharpProfile` so the output dialect (interop attribute, native-int type, generic dispatch strategy) is a generation-time choice rather than hardcoded across the renderers — the enabler for AOT and any future C# dialect (see [ADR-038](docs/adr/038-aot-compilation.md))
