# Supported Features

This is the catalogue of language features the bridge maps across the Kotlin/Native Ôćö C# boundary ÔÇö in **both directions** ÔÇö and how. For the development roadmap (including what is not yet supported) see [ROADMAP.md](ROADMAP.md). For the reasoning behind each mapping follow the ADR links in the last column.

**Direction** (the middle column of every table):

| Glyph | Meaning                                                                               |
|-------|---------------------------------------------------------------------------------------|
| `Ôćĺ`   | Kotlin Ôćĺ C#: a Kotlin feature you author, surfaced as C# (forward, code generation)   |
| `ÔćÉ`   | C# Ôćĺ Kotlin: a feature from a consumed C# NuGet package, surfaced as Kotlin (reverse) |
| `Ôçä`   | both directions                                                                       |

`Ôçä` is the target state. Where a row is one-way, the **Notes** say why ÔÇö either it is **fundamental** (the reverse direction reads a compiled assembly you don't own, so some source-only Kotlin concepts can't round-trip), or it is simply **not built yet** (see [ROADMAP.md](ROADMAP.md)).

**Notes**

- **`[1]`** ÔćÉ The `GCHandle` is released via `Cleaner` (deterministically via `close()` / `use {}`). C# statics land in a `companion object`, instance members on the wrapper. An instance member colliding with the wrapper's own `handle`/`close`/`cleaner` is skipped with a warning. Object return/param nullability is metadata-driven per `NullableAttribute` (`Foo` vs `Foo?`); a new wrapper is created per crossing (identity equality, no caching).
- **`[2]`** ÔćÉ Overloads retain their parameter types and use signature-derived internal bridge identities.
- **`[3]`** Ôćĺ Object methods behave exactly like class methods (e.g. a `String` return Ôćĺ `string`).
- **`[4]`** ÔćÉ Only public top-level, default-`int`, non-`[Flags]` enums with unique contiguous `0..N-1` values. Nullable, nested, sparse, aliased, collection, and generic enum positions are not built yet.
- **`[5]`** Two shapes share one wire format and `data class` surface. **Shape A**: exactly one state-covering constructor whose parameters define the components, in constructor order. **Shape B**: no such constructor (public settable fields and/or `set`/`init` auto-properties); components come from C# declaration order and reconstruct with an object initializer (`new T { A = a, B = b }`) rather than `new T(a, b)`, binding no constructors at all. Alternate public constructors (Shape A only) become Kotlin secondary constructors backed by out-pointer bridge slots. Instance methods and get-only non-component computed properties land on the `data class` (reconstruct-on-call: N component wire args first); static methods land in a `companion object`. A component may itself be a struct at any depth: the wire flattens it recursively, depth-first pre-order (ABI name is the `_`-joined path) while the Kotlin surface stays nested, so structural equality and `copy()` compose across the nesting; all four Shape A/B nesting combinations work, including across a Kotlin package boundary. A member whose flattened ABI arity exceeds Kotlin/Native's 22-argument `CFunction.invoke` ceiling is skipped with a `skipped_abi_arity_limit` diagnostic. Class-typed components, generic structs, structs in collections, alternate constructors, and manual (non-auto) settable properties on a Shape B struct stay deferred.
