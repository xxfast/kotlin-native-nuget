- [054](054-reverse-bridge-registration-observability.md) – Tooling: reverse-bridge registration observability (Proposed): the ADR-041 function-pointer table becomes observable via three parts, deliberately not one. (1) An always-on generated `NugetRegistry` bakes the expected registration-export set in at generation time and records each one as it lands, so the `requireNotNull` guard's message becomes *computed*: "0 of 7 registrations have fired, nothing registered at all, the shims are not compiled into the host assembly (usually a stale `obj/project.assets.json`)" versus "6 of 7 fired, only `Sample.Text.Template` is missing". Those are the two sentences nobody could get an answer to during ADR-053. (2) An always-on registration-time contract self-check: every register export gains two leading scalars (`slotCount: Int`, `contractHash: Long`, both from one shared `RirBridging.kt` function) and its pointer params become nullable; Kotlin refuses to store *any* pointer on mismatch, turning stale-shim/fresh-dylib arity or type drift from silent memory corruption into a named "one of these is stale" error (an amendment to ADR-048's registration contract). (3) An opt-in `NUGET_INTEROP_TRACE=1` registration trace, stderr by default with a `NUGET_INTEROP_TRACEFILE` redirect (the `LD_DEBUG`/`LD_DEBUG_OUTPUT` and `DOTNET_HOST_TRACE`/`DOTNET_HOST_TRACEFILE` shape; xunit 2.9.3 does not capture Console, and a crashed host loses it). Registration granularity only, matching JNI's `-verbose:jni`; per-call tracing, verbosity levels, subsystem categories and an `EventSource` provider all rejected for v1 (EventSource is C#-only, so the Kotlin half of the trace could not participate). Zero cost on the bridge call path
- [055](055-forward-abi-contract-check.md): Tooling (Accepted). Forward C# `CirDllImport` declarations and Kotlin `@CName` exports are generated independently, so collect a normalized wire-level signature from both and fail KSP generation on a missing or mismatched symbol/return/ordered parameter contract. - [056](056-csharp-structs-in-kotlin.md) — Phase 9 C# structs (value types) in Kotlin (Accepted): the ROADMAP's "blittable pass-by-value vs boxed handle" is a false dichotomy — **decompose the struct into its components and never put a struct on the wire**. Pass-by-value is not expressible here (verified: a hand-written `CStructVar` cannot appear in a `CFunction`'s `invoke`, "not a structure or too complex"; only a *cinterop-generated* one can, and ADR-048 generates plain Kotlin source, not a klib), and it covers strictly fewer structs anyway (verified: `[UnmanagedCallersOnly]` blittability is enforced at **runtime**, is **transitive**, and rejects `bool`/`char` with `InvalidProgramException` even nested in a struct — so `readonly record struct Person(string Name, int Age)` could never cross by value). The boxed `GCHandle` is correct but destroys value semantics (`close()`, `Cleaner`, reference equality on a `Point`), the exact trap ADR-014 rejected forward. So: a struct parameter expands to one ABI argument per component, a struct return to `void` + one **out-pointer per component** (verified end-to-end in the real Kotlin-dylib-in-.NET-host topology for `string`/`int`/`bool`/`char`/enum components, both directions), and the Kotlin surface is an immutable `data class` — value equality, `copy()`, no handle, no `close()`, zero registration slots. Partly a mirror of ADR-014 (unwrap-don't-box transfers exactly; arity generalises 1 → N, which is what forces out-pointers). v1 = Shape A structs (one public ctor whose parameters cover all stored state; components = ctor params, matched case-insensitively to properties); new `RirStruct`/`RirStructComponent`/`RirStructType` + `skipped_unsupported_struct`; **`contractHash` must expand struct components** or a struct gaining a field silently changes thunk arity under a stale shim (ADR-054); both generators must share one `abiArgs`/`abiOutArgs` expansion (one `RirParameter` is no longer one ABI argument). Fixes a verified latent bug: the reader currently emits every struct as `{"kind":"class"}`. Unblocks `Nullable<T>` (ADR-053 Decision 3) by fixing its wire format — `byte hasValue` + out-pointer, one crossing, one evaluation — without subsuming it
- [057](057-csharp-overload-sets-in-kotlin.md): Phase 9 C# overload sets in Kotlin (Accepted): preserve ordinary Kotlin method/constructor overloads; derive stable internal thunk/pointer identity from a canonical managed signature plus a checked SHA-256 digest; canonically sort constructor/static/instance/property registration slots instead of trusting MethodDef/declaration order (verified unstable under a scratch declaration-reorder probe); map overloads independently, then fail fast if distinct CLR signatures collapse to one Kotlin signature (ADR-034 mirror); multiple ADR-056 Shape A struct constructors use one unique state-covering constructor for data-class components while alternate constructors get bridge-backed out-pointer slots
- [058](058-csharp-shape-b-structs-in-kotlin.md): Phase 9 C# **Shape B** structs in Kotlin (Accepted): a struct with no state-covering constructor (public fields and/or settable auto-properties) binds by taking its **settable public state** as the component model and reconstructing with an **object initializer** (`new T { A = a, B = b }`). Nothing about ADR-056's wire changes: components in, out-pointers out, immutable `data class`, zero slots when memberless; the Kotlin generator needs no change at all, and the only new C# text is the `{ … }` braces at the two existing `new T(...)` reconstruction sites. Three verified spikes settle the three concerns ADR-056 deferred on. (1) *Field extraction*: an auto-property's backing field is `Private` + `[CompilerGenerated]` (so an accessibility filter alone excludes it), and `GetFields()` yields declaration order **interleaving** real fields with backing fields, which is what makes one unified component order recoverable in a single pass. Coverage is therefore *structural*, not a count: every non-static field must be either a settable public field or the backing field of a settable public auto-property, else the struct is skipped (a count-based rule silently drops state, and cannot order a manual property anyway). (2) *The `init` landmine is defused by the choice of syntax, not by detecting it*: an object initializer is exactly the context where an `init` accessor is callable, cross-assembly (verified), so `set` and `init` are both just "settable" and no `IsExternalInit` detection is needed. (3) *Order* is the real cost: FieldDef declaration order becomes the `data class` primary-constructor order, and a C# field reorder is source-compatible for C# callers but silently reorders positional Kotlin construction (alphabetical sorting rejected: it would make the Kotlin order permanently disagree with the C# declaration, trading a rare hazard for an always-armed one; the defence is generated KDoc plus named arguments). Also **amends ADR-056 Decision 3a rule 3** (a constructor parameter may now match a public *field*, not only a property) because otherwise the ordinary `struct { public int A; public int B; public S(int a, int b) }` falls to Shape B, its ctor becomes an ADR-057 alternate, and its Kotlin signature collides with the generated primary constructor, hard-failing the build; and **fixes a latent ADR-054 hole** (`signaturePart()` expands a struct *reference* to component types only, so swapping two same-typed components is invisible to the stale-shim contract check, which Shape B promotes from theoretical to plausible). Alternate constructors on Shape B structs deferred (same collision), manual non-auto settable properties deferred (coverage unprovable from metadata); nested-struct and class-typed components stay deferred
- [059](059-nested-struct-components-in-kotlin.md): Phase 9 **nested struct components** in Kotlin (Accepted): a struct-typed component is **flattened recursively, depth-first, pre-order**, so the flat argument list is the leaves of the component tree; the Kotlin surface **stays nested** (`val mother: Profile`, a `data class` inside a `data class`, with deep structural equality and composing `copy()`); the C# side **reconstructs recursively**, each struct using its own ADR-058 shape, so the four A/B nesting combinations need no rules of their own (all four verified round-tripping through a real C-ABI function pointer, at depth 3, with a `string` leaf inside the nested struct). The RIR keeps the struct **reference** and does **not** pre-flatten (verified: `RirStructComponent.type` is already a `RirTypeRef` admitting `RirStructType`, so **the JSON schema does not change at all**; pre-flattening would destroy the nesting the Kotlin surface needs and make `readName` a lie), and `signaturePart()` **already recurses**, so ADR-054's contract hash covers nesting with no code change. Termination is verified, not assumed: **CS0523** rejects direct, mutual, generic-mediated and `Nullable<T>`-mediated struct cycles (a cycle through a class/array/static field is legal C# but is not a component, and a handle component would be a wire *leaf* anyway), and the reader classifies by **fixed-point iteration** so it cannot recurse on a cyclic IL assembly the C# compiler never saw. The one genuinely new constraint, and the reason this is an ADR: **Kotlin/Native's `CFunction.invoke` accepts at most 22 arguments** (verified: 23 fails with "none of the following candidates is applicable", at the *call site*, mentioning neither arity nor structs), a ceiling nothing has been close to before and that flattening makes reachable with ordinary types (two 8-leaf structs in and one out is 24), so an over-arity member is **skipped with a `skipped_abi_arity_limit` diagnostic from the shared registrable filter**, never left for the Kotlin compiler to reject inside generated code. Also fixes a verified diagnostic bug: today a nested struct is skipped with a message naming the *wrong rule* (a Shape B auto-property rule for a Shape A struct; "not a bridgeable bound **class**" for a struct), because `ExtractStruct` discards the Shape A reason when it falls through to Shape B
- [060](060-adversarial-forward-fixture.md): Tooling — the adversarial forward fixture, one red seam + two green guards (Accepted; steps 1–3 shipped): the mirror of `Test.Household` is **not one artefact**, because the forward direction has three gates where the reverse has one (generated Kotlin compiled by `:test-library:compileKotlinMingwX64`; generated C# compiled *inside every consumer* via `contentFiles buildAction="Compile"`; and `ForwardAbiContract` at generation time). **Every defect cell would break a build rather than fail an assertion** if simply added to `test-library` — twelve emit invalid Kotlin (no `.nupkg`, nothing downstream runs), one emits `CS0111` inside `Interop.cs` (every consumer fails), five are valid-but-lying C# whose *shape* can only be asserted at a compile-time call site (`CS0029`/`CS1503`, hard errors regardless of warning settings), and one silently vanishes (`CS1061`); no cell's red is free — an earlier draft thought `Char` could be red at runtime, but it leaks `IntPtr` into the *public* signature (`string Tag(IntPtr)`), so it is a call-site defect too. Driving the **real** processor (KSP2's own `KotlinSymbolProcessing` entry point — **no kctfork**; 2.4s first run, then ~130ms) and the **real** konanc settled the design further than the alternatives describe: **all red lives in Tier 1** as strict-`@XFail` JUnit tests (an unexpected pass **fails the build**, so a fix cannot land leaving its marker behind), while Tiers 2 and 3 become **regression** tiers landing green *with* each fix. Consequently `scripts/verify.sh` stays green on a tree with all twelve defects and needs no change: the **expected-errors manifest is rejected** (inverting `GeneratedBindingsCheck`'s exit code would blind the gate to *unrelated* regressions — the kind it caught on day one — for the whole xfail period; a red CI is ignored, a lying one is believed), and **Tier 1 must not compile C#** (it would make `nuget-processor`'s unit tests need the .NET SDK, eroding MVP.md line 62's stated property that the forward direction needs none; a structural assertion on the generated `Interop.cs` carries the red just as well). Verified both unverified reports: **BUG-010 is real** (all four reported failure modes reproduced) and **BUG-008 is real but mis-framed** (it is not a type bug but a *position* bug). - [061](061-method-return-marshalling.md): Forward — non-primitive and nullable return marshalling for class methods and extension functions (Accepted): route both method-return positions through the same property-getter cascade (object → `StableRef` carrier + `new T(handle)`; `List<T>` → object carrier materialized by the shared `NugetListNative` helpers, which need `needsListSupport` widened to also scan method/extension returns; nullable object → `IntPtr.Zero`-sentinel; nullable `String` → single-call `PtrToStringUTF8`), and resolve the one genuine decision — a nullable **primitive** method return cannot reuse ADR-002's two-call `hasValue`+`value` pattern because a method with side effects must not be invoked twice — with a **single-call out-parameter**: the export returns `Boolean` has-value and writes the value through a `valueOut: COpaquePointer?` (mirror of ADR-024's `errorOut` out-write), C# reads `Native_X(_handle, out int value, out IntPtr error)`. Boxing (the ObjC/Swift `KotlinInt` / JNI `Integer` precedent) considered and rejected for its `StableRef` alloc + second unbox crossing. Closes ROADMAP L61 return positions + ADR-060 cells 4/5/6. Load-bearing inferred (not spiked): `reinterpret<IntVar>().pointed.value` round-trips on the real mingwX64 toolchain — the walking-skeleton integration test must confirm before relying on it; `Boolean?`/`Char?` returns and nullable **parameters** deferred as separate items
- [063](063-forward-declaration-level-export-scoping.md): Forward, declaration-level export scoping (MF-001, Accepted): the processor bridges every public declaration (`resolver.getAllFiles()`, no filter) because there is no export set; `rootPackage` only renames the C# namespace (`mapPackageToNamespace`), it does not scope. Adds an explicit **package include/exclude filter in `publish {}`**: `include(vararg)`/`exclude(vararg)` on `NugetPublishConfig`, mirroring the reverse `bind { include/exclude }` (ADR-044/047) verb-for-verb and prefix predicate exactly (exclude wins; empty include = all; `pkg == p || pkg.startsWith("$p.")`). Threaded through the existing `rootPackage` path (three KSP options → three `NugetContext` fields, the third being `boundPackages`) and applied at the single `allDeclarations` choke point, so every downstream category is scoped for free. Default: an empty `include` falls back to `rootPackage` when set, else all packages, so `rootPackage` is now a scoping knob and not only a renaming one (a deliberate, documented behaviour change). A reverse-bound package (from a module's own `bind {}`) is always exported regardless of `include`/`exclude`, checked first in the predicate: `test-library`'s own forward/reverse round trip broke on the first cut of this feature when a bound package was dropped from scope. Rejected: a per-declaration annotation (opt-in contradicts GOALS #1 plug-and-play like the already-rejected `@ExperimentalNugetApi`; opt-out doesn't isolate unrelated same-module API), deferred as a future `@HiddenFromObjC`-style layer; a full reachability closure now (Alt 4, the MF-001/MF-003-unified framing) is rejected because MF-003 is cut from MVP and sequenced *after* forward diagnostics + the value-class-parameter fix, so building it now re-arms the constructs module isolation hides. This ADR **chooses the roots-are-packages framing** so MF-003 generalises the include set into seed roots rather than replacing it. Filter runs before the `ForwardCallablePlanner`, so an out-of-scope declaration is silently not bridged and never triggers the forward "unsupported" diagnostic.
